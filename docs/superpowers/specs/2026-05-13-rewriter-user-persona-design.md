# Query Rewriter User-Persona 강화 설계

## 배경

`query_rewriter`는 사용자 입력을 시간 표현 해석, 대화 맥락 보충, 용어 사전 치환을 거쳐 재작성한 다음 후속 노드(router, subagent)에 넘긴다. 출력은 **항상 사용자가 에이전트에게 보내는 발화 형태**여야 한다 — 다음 노드(router)가 이를 사용자 의도로 해석해 라우팅하기 때문이다.

현재 시스템 프롬프트(`src/query_rewriter/prompt.py:146-210`)는 이 페르소나 제약을 명시하지 않는다. 결과적으로 짧은 명사구 단답이 chat_history 맥락 없이 들어오면 LLM이 "에이전트가 사용자에게 답하는 톤"으로 출력하는 표류가 관찰된다.

## 재현된 실패 모드

`scripts/repro_rewriter_bug.py`로 측정 (5회 반복).

| 케이스 | chat_history | 출력 (대표) |
|---|---|---|
| 단발 "고객지원" | 없음 | "고객 지원에 대한 정보를 요청합니다." (5/5 응답-형식) |
| 멀티턴 "고객지원" | 부서 안내까지 4개 | "고객지원 부서의 평균 급여를 알려주세요." (5/5 정상) |
| 대조군 "상품 재고 알려줘" | 없음 | "상품 재고를 확인해 주세요." (3/3 정상) |

핵심: chat_history가 있으면 정상 동작한다. 문제는 **chat_history가 비어 있거나 부족한 상태에서 짧은 명사구 입력**이 들어오면 응답자 어투로 빠지는 것이다.

## 원인

프롬프트에서 빠진 두 가지 제약:

1. **출력 페르소나 미정의** — "재작성된 쿼리만 출력하세요"라는 형식 지시는 있으나, 출력이 *사용자 입장의 질문/요청 형태*여야 한다는 제약이 없다. LLM이 짧은 명사를 보고 "내가 답할 차례"로 해석해도 프롬프트상 위반이 아니다.
2. **짧은 명사구/단답형 입력 처리 규칙 부재** — 기존 §2의 "응/네/그래" 확인 응답 규칙은 있으나, "고객지원", "엔지니어링" 같은 명사-단답에 대응하는 변환 규칙은 없다. chat_history가 있어도 우연히 잘 풀리는 정도(LLM 자체 추론에 의존).

도입부의 "사용자 쿼리를 명확하게 재작성하는 전처리기"는 정체성을 한 줄로만 설정해 표류를 막기에 약하다.

## 목표

리라이터 출력이 항상 사용자 입장의 질문/요청 형태로 유지되도록 프롬프트를 강화한다. 코드 구조와 외부 시그니처는 변경하지 않는다.

비목표:

- LLM 컴플라이언스 실패에 대비한 정규식 후처리 가드 (false positive 위험, 유지보수 부담으로 제외)
- 출력의 JSON 구조화 (ROI 낮음, 파이프라인 영향 큼)
- 별건인 "전체 상품 재고" 자동 보충 이슈는 분리된 spec/plan으로 추후 진행

## 프롬프트 변경 명세

### 1) 도입부 페르소나 강화

기존:

> 당신은 사용자 쿼리를 명확하게 재작성하는 전처리기입니다.

변경:

> 당신은 사용자 쿼리를 명확하게 재작성하는 **전처리기**입니다.
> 출력은 반드시 **사용자가 에이전트에게 보내는 질문 또는 요청** 형태여야 합니다.
> 당신은 답변자가 아닙니다. "안내드립니다", "알려드립니다", "정보를 제공합니다", "~입니다" 같은 응답자 어투를 절대 사용하지 마세요. 사용자의 입장에서 묻거나 요청하는 문장으로만 작성하세요.

### 2) §2 대화 맥락 보충 절에 명사구 단답 규칙 추가

기존 불릿 리스트(이거/그거, 더 해줘, 반대로 해줘, 응/네/그래) 다음에 추가:

> - **명사구 단답** (예: "고객지원", "엔지니어링", "3월", "취소", "5번 상품") → chat_history에서 직전 에이전트 메시지가 어떤 변수 또는 선택지를 물었는지 찾아 완전한 질문 형태로 풀어 쓴다.
>   - 예: 직전 에이전트가 "무슨 부서가 궁금하세요?"였고 사용자 입력이 "고객지원"이면 → "고객지원 부서의 평균 급여를 알려줘"
>   - 직전 맥락이 없거나 단답을 매칭할 질문이 없으면 "<단답>에 대해 알려줘" 형태의 일반 질문으로 변환한다. 응답자 어투로 평서문을 생성하지 않는다.

### 3) 출력 형식 절 강화

기존:

> 재작성된 쿼리만 출력하세요. 설명이나 부가 텍스트를 포함하지 마세요.

변경:

> 재작성된 쿼리만 출력하세요. 설명, 인사말, "안내드립니다" 같은 응답자 어투는 포함하지 마세요. **반드시 사용자가 에이전트에게 보내는 질문 또는 요청 형태**로만 출력하세요. 평서문/안내문/답변문 형태로 출력하면 안 됩니다.

## 테스트 추가

`tests/test_query_rewriter.py`의 `TestPromptConfirmationRule`과 동일한 패턴으로 프롬프트 빌더 출력에 대한 어서션 테스트 3개 추가:

1. **`test_prompt_forbids_response_tone`** — `build_rewriter_system_prompt(now)` 결과에 "안내드립니다" 같은 금지 표현이 포함된 금지 규칙이 들어있는지 확인.
2. **`test_prompt_includes_short_fragment_rule`** — 명사구 단답 처리 규칙(키워드: "명사구 단답" 또는 "단답") 포함 여부 확인.
3. **`test_prompt_includes_user_persona_constraint`** — 사용자 입장 페르소나 제약 어구(키워드: "사용자가 에이전트에게 보내는" 또는 동등 표현) 포함 여부 확인.

세 케이스 모두 LLM 호출 없이 프롬프트 빌더 텍스트를 검증한다(기존 단위 테스트 정책과 일치).

## 회귀 검증

구현 완료 후 `scripts/repro_rewriter_bug.py` 재실행. 기대값:

- **CASE 1** ("고객지원" 단발, chat_history 없음): 응답-형식 → 사용자 질문 형태 (예: "고객지원에 대해 알려줘"). "안내드립니다"/"정보를 요청합니다" 같은 응답자 어투 출현 0/5.
- **CASE 2** ("고객지원" + 멀티턴): 그대로 "고객지원 부서의 평균 급여를 알려주세요." 류 유지 (5/5).
- **CASE 3** (대조군 "상품 재고 알려줘"): 그대로 "상품 재고를 확인해 주세요." 류 유지 (3/3).

LLM-as-Judge 기반 정량 평가는 본 spec의 범위 외(추후 evals/test_cases.yaml에 케이스 추가 검토).

## 위험과 완화

- **LLM 컴플라이언스 의존**: 프롬프트만으로는 100% 차단이 어렵다. `scripts/repro_rewriter_bug.py`를 회귀 게이트로 사용하고, 향후 실패가 재현되면 evals 케이스로 승격해 정기 측정한다.
- **기존 테스트 영향 없음**: `TestPromptConfirmationRule` 등 기존 어서션은 새 프롬프트에서도 그대로 성립한다(확인 응답 규칙은 유지).
